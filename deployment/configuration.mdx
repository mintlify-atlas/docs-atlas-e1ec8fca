## Configuration Overview

OSPADEP uses a centralized configuration system in `config.js` that adapts based on the environment (development vs production).

## Environment Configuration

### Development vs Production

The application automatically adjusts settings based on the `ENVIRONMENT` variable:

<CodeGroup>
```javascript Development
const CONFIG = {
    ENVIRONMENT: 'development',
    ADMIN_PASSWORD_HASH: null, // Uses simple password
    SESSION_TIMEOUT: 120, // 2 hours for convenience
    MAX_LOGIN_ATTEMPTS: 10 // More lenient
};
```

```javascript Production
const CONFIG = {
    ENVIRONMENT: 'production',
    ADMIN_PASSWORD_HASH: 'sha256_hash_here',
    SESSION_TIMEOUT: 30, // 30 minutes
    MAX_LOGIN_ATTEMPTS: 3 // Strict security
};
```
</CodeGroup>

### Environment Detection

The `getConfig()` function automatically applies environment-specific overrides:

```javascript
function getConfig() {
    if (CONFIG.ENVIRONMENT === 'development') {
        return {
            ...CONFIG,
            ADMIN_PASSWORD_HASH: null,
            SESSION_TIMEOUT: 120,
            MAX_LOGIN_ATTEMPTS: 10
        };
    }
    return CONFIG;
}
```

## Core Configuration Options

### Security Settings

<ParamField path="ENVIRONMENT" type="string" default="development" required>
  Application environment mode.
  
  **Options:**
  - `development` - Local development with relaxed security
  - `production` - Production deployment with strict security
  
  **Example:**
  ```javascript
  ENVIRONMENT: 'production'
  ```
</ParamField>

<ParamField path="ADMIN_PASSWORD_HASH" type="string" required>
  SHA-256 hash of the admin password. Never store plain text passwords.
  
  **Default:** `a665a45920422f9d417e4867efdc4fb8a04a1f3fff1fa07e998e86f7f7a27ae3` (hash of "hello")
  
  **Generate:**
  ```bash
  echo -n "your_password" | sha256sum
  ```
  
  <Warning>
  Change this immediately before production deployment!
  </Warning>
</ParamField>

<ParamField path="SUPER_ADMIN_PASSWORD" type="string">
  Super admin password for elevated privileges (future feature).
  
  **Default:** `superadmin2024`
  
  <Warning>
  This should also be hashed in production. Currently stored as plain text for development.
  </Warning>
</ParamField>

<ParamField path="SESSION_TIMEOUT" type="number" default="30">
  Session timeout duration in minutes.
  
  **Recommended values:**
  - Development: `120` (2 hours)
  - Production: `30` (30 minutes)
  - High security: `15` (15 minutes)
  
  **Example:**
  ```javascript
  SESSION_TIMEOUT: 30
  ```
</ParamField>

<ParamField path="MAX_LOGIN_ATTEMPTS" type="number" default="3">
  Maximum failed login attempts before account lockout.
  
  **Recommended:** `3` attempts (industry standard)
  
  After reaching this limit, the account is locked for 15 minutes.
</ParamField>

### User Roles

<ParamField path="USER_ROLES" type="object">
  Role definitions for future multi-user support.
  
  **Structure:**
  ```javascript
  USER_ROLES: {
      SUPER_ADMIN: 'super_admin',
      ADMIN: 'admin',
      USER: 'user'
  }
  ```
  
  <Note>
  Currently, only admin role is implemented. Multi-user support planned for future releases.
  </Note>
</ParamField>

### Synchronization Settings

<ParamField path="SYNC_INTERVAL" type="number" default="5000">
  Interval for data synchronization in milliseconds.
  
  **Default:** `5000` (5 seconds)
  
  Used for future backend sync functionality.
</ParamField>

<ParamField path="AUTO_SYNC" type="boolean" default="true">
  Enable automatic data synchronization.
  
  **Example:**
  ```javascript
  AUTO_SYNC: true
  ```
  
  <Note>
  Requires backend API implementation (Option 2 deployment).
  </Note>
</ParamField>

### Audit & Logging

<ParamField path="LOG_ALL_ACTIONS" type="boolean" default="true">
  Enable comprehensive action logging for security and audit purposes.
  
  **Logs include:**
  - Login attempts (success/failure)
  - Price changes
  - Admin actions
  - Security incidents
</ParamField>

<ParamField path="MAX_AUDIT_RECORDS" type="number" default="1000">
  Maximum number of audit records to retain in localStorage.
  
  **Recommended:** `1000` records
  
  Older records are automatically pruned when limit is reached.
</ParamField>

### Data Management

<ParamField path="BACKUP_INTERVAL" type="number" default="24">
  Automatic backup interval in hours.
  
  **Default:** `24` hours
  
  <Note>
  Backup functionality requires backend implementation (Option 2).
  </Note>
</ParamField>

<ParamField path="MAX_HISTORY_RECORDS" type="number" default="100">
  Maximum price change history records to retain.
  
  **Example:**
  ```javascript
  MAX_HISTORY_RECORDS: 100
  ```
</ParamField>

## API Configuration

API endpoints for future backend integration:

<ParamField path="API_BASE_URL" type="string" default="''">
  Base URL for backend API.
  
  **Examples:**
  - Development: `http://localhost:3000`
  - Production: `https://api.ospadep.com`
  
  **Current:** Empty (no backend)
  
  ```javascript
  API_BASE_URL: '' // Disabled
  ```
</ParamField>

<ParamField path="BACKUP_ENDPOINT" type="string" default="'/api/backup'">
  Endpoint for backup operations.
  
  **Full URL:** `{API_BASE_URL}/api/backup`
</ParamField>

<ParamField path="PRICES_ENDPOINT" type="string" default="'/api/prices'">
  Endpoint for price data operations.
  
  **Methods:**
  - `GET /api/prices` - Fetch current prices
  - `POST /api/prices` - Update prices
  - `PUT /api/prices/:id` - Update specific price
</ParamField>

<ParamField path="SYNC_ENDPOINT" type="string" default="'/api/sync'">
  Endpoint for data synchronization.
  
  Used for multi-client sync in future backend implementation.
</ParamField>

<ParamField path="AUDIT_ENDPOINT" type="string" default="'/api/audit'">
  Endpoint for submitting audit logs to backend.
  
  **Payload:**
  ```javascript
  {
      type: 'LOGIN_SUCCESS',
      timestamp: '2026-03-05T10:30:00Z',
      data: { /* event details */ }
  }
  ```
</ParamField>

## Environment Variables

### Using Environment Variables

For better security, use environment variables instead of hardcoding values:

```javascript
const CONFIG = {
    ENVIRONMENT: process.env.NODE_ENV || 'development',
    ADMIN_PASSWORD_HASH: process.env.ADMIN_PASSWORD_HASH,
    API_BASE_URL: process.env.API_BASE_URL || '',
    ADMIN_EMAIL: process.env.ADMIN_EMAIL || '',
    SLACK_WEBHOOK: process.env.SLACK_WEBHOOK || ''
};
```

## Configuration Best Practices

### Security

1. **Never commit secrets** to version control
2. **Use environment variables** for sensitive data
3. **Rotate passwords** regularly (every 90 days)
4. **Hash all passwords** using SHA-256 or better
5. **Enable logging** in production

### Performance

1. **Set appropriate timeouts** based on usage patterns
2. **Limit audit records** to prevent localStorage overflow
3. **Configure sync intervals** based on update frequency

### Maintenance

1. **Document custom settings** in your deployment notes
2. **Keep backups** of configuration files
3. **Test configuration** changes in development first
4. **Monitor logs** for configuration issues

## Troubleshooting

### Configuration Not Loading

**Issue:** `APP_CONFIG` is undefined

**Solution:**
```javascript
// Ensure config.js is loaded before other scripts
<script src="config.js"></script>
<script src="security.js"></script>
<script src="app.js"></script>
```

### Environment Not Detected

**Issue:** Production behaves like development

**Solution:** Check `ENVIRONMENT` value:
```javascript
console.log(APP_CONFIG.ENVIRONMENT); // Should be 'production'
```

### Session Timeout Not Working

**Issue:** Sessions never expire

**Solution:** Verify SecurityManager is instantiated:
```javascript
console.log(window.SecurityManager); // Should be defined
console.log(APP_CONFIG.SESSION_TIMEOUT); // Should be 30
```
